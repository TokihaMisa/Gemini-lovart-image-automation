# ChatGPT 浏览器模式设计

## 目标

在现有提示词生成来源中增加 `gpt_browser`。该模式自动操作 `https://chatgpt.com`，复用项目现有的持久化浏览器登录档案，并为每个商品执行独立的临时对话流程：新聊天、开启临时对话、发送前置提示、上传商品与参考图片、发送商品提示词、等待并提取回复。

ChatGPT 使用账号当前的默认模型，自动化流程不操作模型选择器。

## 范围

本次包含：

- CLI 的 `--prompt-source gpt_browser` 选项和交互式来源选择。
- WebUI 的 `gpt_browser` 提示词引擎选项。
- ChatGPT 页面自动化、上传等待、回复等待、回复提取和调试快照。
- 将 ChatGPT 客户端接入现有商品处理与 Lovart 确认判断流程。
- 配置示例、使用文档和自动化测试。

本次不包含：

- OpenAI API 模式。
- 自动选择或切换 ChatGPT 模型。
- 自动填写账号、密码或处理登录验证。
- 重构现有输出状态和历史 `gemini_*` 兼容字段。

## 架构

新增独立的 `ChatGPTBot` 页面适配器。它实现现有商品管线依赖的两个公开方法：

- `generate_prompt(...)`
- `advise_lovart_confirmation(...)`

Gemini 和 ChatGPT 各自维护页面选择器、会话初始化、上传状态识别、生成状态识别和回复提取逻辑。浏览器可执行文件解析、持久化上下文创建、僵尸进程清理和商品处理管线继续由 `main.py` 统一协调。

这种边界避免把两个经常变化的网站 DOM 规则塞入同一个机器人类，同时不复制商品处理和 Lovart 调用逻辑。

## 配置和入口

配置新增：

```yaml
chatgpt:
  base_url: "https://chatgpt.com"
  reply_timeout: 300
  upload_timeout: 120
  upload_attempts: 3
```

`browser.user_data_dir` 继续作为默认持久化档案目录，因此 Gemini 与 ChatGPT 复用同一浏览器登录档案。CLI 和 WebUI 都将 `gpt_browser` 作为独立提示词来源。

浏览器启动函数接收提示词来源，根据来源选择目标 URL、登录检测提示和机器人类。现有 `gemini_browser` 行为保持不变。

## 每商品数据流

1. 导航到 ChatGPT 首页。
2. 点击“新聊天”，并确认进入一个新的空白会话。
3. 开启“临时对话”；如果无法确认临时状态，停止当前商品，避免内容进入普通历史会话。
4. 读取并发送现有 `preamble.txt`，等待回复完成。
5. 上传该商品的图片，等待附件达到预期数量且上传状态稳定。
6. 使用现有 `build_design_prompt(...)` 构建并发送商品提示词。
7. 等待新的助手回复完成，提取最后一条助手消息。
8. 将结果交回现有 Lovart 管线。

后续 Lovart 出现确认问题时，`ChatGPTBot.advise_lovart_confirmation(...)` 在当前临时会话中发送现有确认判断提示，并解析回复。这样与 Gemini 浏览器模式的业务行为一致。

## 页面自动化策略

页面控件采用分层识别：

1. 优先使用稳定的语义属性，如 `data-testid`、`aria-label` 和角色。
2. 兼容中文与英文界面文本。
3. 必要时使用可见元素 DOM 扫描作为后备方案。

上传优先直接设置页面中的文件输入；若不可用，再点击附件按钮并处理文件选择器。发送文本优先定位当前可见的消息编辑框和发送按钮。

回复只从助手消息容器提取，不使用整个页面正文作为正常成功路径，避免侧边栏和历史聊天文本混入生成结果。

## 错误处理

以下情况使当前商品失败：

- 无法进入新聊天。
- 无法开启或确认临时对话。
- 文件上传未在超时内完成。
- 回复未出现、持续生成超时或提取结果过短。

失败时在当前运行目录的 `browser-debug/<商品ID>/` 保存截图和 HTML，并使用包含阶段信息的标签命名。错误日志使用 `ChatGPT:` 前缀。

登录页或认证跳转属于运行前阻塞：交互运行时提示用户手动登录后继续；非交互运行时明确失败，不尝试自动输入凭据。

## 输出兼容性

现有商品管线、结果汇总和 Lovart 调用继续使用原有接口。为避免一次功能增加演变为数据迁移，本次保留 `gemini_prompt.txt`、`gemini_chars` 和 `gemini_done` 等历史兼容名称；运行日志会明确记录实际来源为 ChatGPT。

未来如需统一命名，可单独设计 `prompt_source`、`prompt_chars` 和 `prompt_result.txt` 的兼容迁移。

## 测试和验收

自动化测试覆盖：

- CLI 和 WebUI 暴露 `gpt_browser`。
- 浏览器启动器按来源选择 ChatGPT URL 和 `ChatGPTBot`。
- 每商品严格按“新聊天 → 临时对话 → preamble → 上传 → 商品提示 → 回复”执行。
- 临时对话无法确认时不会继续发送内容。
- 上传完成检测和重试。
- 只提取最后一条助手回复。
- 错误时生成截图和 HTML 快照。
- ChatGPT 可执行 Lovart 确认判断。
- 现有 Gemini、API、NVIDIA 和 Lovart 测试不回归。

验收标准：在已登录 ChatGPT 的 `browser_profile` 下选择 `gpt_browser`，每个商品都产生新的临时对话，图片和提示词发送成功，返回的提示词继续进入 Lovart 流程；任一关键会话控制失败时不会污染普通聊天历史。
